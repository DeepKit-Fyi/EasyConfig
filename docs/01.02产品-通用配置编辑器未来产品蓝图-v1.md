# 01.02 通用配置编辑器未来产品蓝图

## 1. 愿景

在 MVP 的基础上，通用配置编辑器的长期愿景是：

> 面向桌面与服务端应用的一体化「配置建模与治理平台」，
> 既能支持本地文件编辑，又能逐步承载环境管理、版本管理、团队协作等能力，
> 但始终保持“**只负责配置，不接管业务逻辑**”的边界。

从演进路径上，可以分为：

- **v1.x：本地专业版** – 打磨编辑体验、验证能力和类型系统，适配更多复杂配置场景；
- **v2.x：团队协作版** – 引入基础的环境管理、差异比较、历史版本与简单协作；
- **v3.x：配置治理平台雏形** – 考虑与远程配置中心/云服务打通，但仍保持「编辑器」角色。

---

## 2. 中短期演进（v1.x ~ v2.x）

### 2.1 类型系统与编辑器扩展

在现有 `TConfigType` + `TEditorType` 的基础上：

- 补齐/增强复杂类型：
  - 工作流、调度、权限、I18n、单位换算、版本控制等已经在 `attribute.md` 中列出的类型；
  - 对应实现可视化编辑器 Frame，并通过 `ControllerConfigs` 统一管理。
- 提供「**可扩展类型**」机制：
  - 在 `UtilsTypes.pas` 里预留 `ctCustom` / `etCustom`；
  - 允许第三方在不改核心代码的前提下注册新的类型与编辑器。

### 2.2 验证系统增强

- 将 `ConfigValidator` 从“局部辅助工具”提升为**统一的验证管线**：
  - 支持针对单个字段的即时验证（如编辑框失焦时提示）；
  - 支持针对整个配置单元的批量验证（保存前 / 专门的“验证”按钮）。
- 增加更丰富的规则类型：
  - 交叉字段校验（如开始时间 < 结束时间；目录必须存在且不同于另一个目录）；
  - 引用校验（`_ref` 所指 `_id` 必须存在且类型匹配）；
  - 环境约束（某些字段只在特定环境生效）。
- 提供验证规则可视化/配置化支持：
  - 将部分规则抽象到 JSON/INI 配置中，而不是全部硬编码在代码里。

### 2.3 搜索、过滤与导航

- 在树/网格中支持：
  - 关键字搜索字段名/描述/值；
  - 过滤显示“有问题的字段”、“最近修改的字段”、“某一类类型的字段”；
  - 高亮当前搜索结果并支持前后跳转。
- 为复杂 JSON 结构提供“路径导航栏”（类似 breadcrumb），方便在深层对象间跳转。

### 2.4 差异比较与合并

- 支持比较两份配置单元（A.ini + A.json vs B.ini + B.json）：
  - 展示新增/删除/修改的键和值；
  - 可按节/路径分组查看 diff；
  - 允许单向合并某些差异。
- 为**多环境配置**（dev/stage/prod）提供对比视图：
  - 例如：同一个“逻辑配置项”在不同环境下的值对比；
  - 高亮环境间不一致或缺失项。

### 2.5 历史版本与回滚

- 提供轻量级本地历史：
  - 每次保存自动（或按设置）生成带时间戳的快照（如 `.bak.yyyyMMddHHmmss`）；
  - 在 UI 中可以浏览某个配置文件的历史版本列表；
  - 支持从历史版本恢复当前文件，或做“当前 vs 某一历史”的 diff。

### 2.6 模板与向导

- 在 `config/*.template-*.json` 的基础上：
  - 引入“配置向导”：选择模板 → 填写若干关键字段 → 自动生成一套基础配置；
  - 不改变编辑器“通用工具”的定位，只是降低新项目上手成本。

### 2.7 更友好的错误处理与诊断

- 结合 `ErrorLogger` / `ErrorDialog` / `ExceptionRecovery`：
  - 保存失败、解析失败、验证失败时给出更具体的信息与建议；
  - 提供“自动修复并备份原文件”的选项；
  - 当 JSON 结构严重损坏时，尽量给出“可以承受的最小载入结果”。

## 3. 远期方向（v3.x 及以后）

### 3.1 多环境与配置集管理

- 将“单一 INI+JSON 对”提升为“**配置集合**”概念：
  - 支持按项目/模块管理多个配置单元；
  - 支持为每个配置定义多个环境变体（dev / test / stage / prod）；
  - 引入「配置集」级别的验证和 diff。

### 3.2 远程配置/云端协作（可选模块）

在保持“**编辑器本体不依赖服务器**”的前提下，增加可选的远程能力：

- 通过插件或扩展模块接入：
  - Git 仓库（把配置作为代码管理）；
  - 远程配置中心（如自建 HTTP API）；
  - 云存储（同步配置文件到云盘）。
- 编辑器仍然只负责：
  - 拉取配置到本地；
  - 提供结构化编辑和验证；
  - 将修改推送回远程；
  - 不负责发布/灰度/滚动升级等运维流程。

### 3.3 插件与脚本化扩展

- 提供插件系统：
  - 可以动态注册新的 `TConfigType` / `TEditorType`；
  - 可以添加新的验证规则、导入导出格式；
  - 可以注册自定义的“配置检查器”（如安全扫描、依赖检查）。
- 提供轻量级脚本 API：
  - 允许使用某种脚本语言（如 PascalScript、Lua 或 JS）批量修改配置；
  - 提供统一的“预处理/后处理 hook”（保存前执行脚本、验证后执行脚本）。

### 3.4 更强的可视化能力

- 更丰富的 UI 预览：
  - 在不嵌入整个业务程序的前提下，针对部分领域提供“近似预览”（如布局示意图、颜色+字体主题预览）；
  - 保持边界：预览只基于配置，不运行真实业务逻辑。
- 交互式图编辑：
  - 对于工作流、调度、权限树等复杂结构，可使用拖拽式图形界面进行编辑。

### 3.5 多语言支持与跨平台

- 在界面上支持中英等多语言切换；
- 在技术上，长期可以考虑将验证逻辑、类型系统和部分 UI 逻辑抽象成与 VCL 无关的核心库，为未来跨平台 UI（如 FMX / Web 前端）打基础。

## 4. 永久边界：未来也「不会做」什么

无论产品如何演进，通用配置编辑器仍然保持以下清晰边界：

### 4.1 不取代业务系统

- 不代替数据库管理工具（如 SQL Server Management Studio、phpMyAdmin）；
- 不代替 API 网关或监控平台；
- 不代替 CI/CD 或部署平台。

配置编辑器的角色是：

> 为这些系统提供高质量的、经过验证的配置输入，
> 而不是接管它们的运行、监控或运维职责。

### 4.2 不管理运行时生命周期

- 不负责启动/停止业务进程，不是服务管理器；
- 不长期驻留后台监听变更、主动推送给所有客户端（这类能力属于配置中心/服务治理范畴）。

### 4.3 不实现复杂安全与合规体系

- 不内置完整的 RBAC、审计追踪、审批流；
- 更适合作为「单机/小团队工具」或「被嵌入到更大平台中的一块配置编辑组件」。

未来如有“企业级”需要，应通过与现有安全/审计基础设施集成来实现，而不是在本工具内重造轮子。

## 5. 与现有代码结构的衔接建议

为了让未来蓝图可落地，建议在当前/后续开发中遵循一些约束：

1. **单一事实来源（Single Source of Truth）**：
   - 所有配置类型相关定义集中在 `UtilsTypes.pas`；
   - 所有关于 INI+JSON 结构和规则的说明集中在 `config/ini_json_rules.md` 等文档中。

2. **UI 与模型解耦**：
   - 复杂属性的 UI 统一通过 `TBaseConfigFrame` + `ControllerConfigs` 接入；
   - 验证逻辑尽量收敛在 `ConfigValidator` 及其配置中。

3. **测试前置**：
   - 为新的类型、验证规则、加载/保存逻辑增加单元测试（复用 `backup` 目录中的测试框架）；
   - 优先为“差异比较”、“历史版本”等高风险功能增加回归测试。

4. **脚本/一次性修复工具分层**：
   - 把目前根目录下的批量修复脚本收敛到 `tools/` 或类似目录，标注用途与风险；
   - 保证日常开发流程尽量不依赖这些“一次性脚本”，而是依赖稳定 API 与工具链。

通过上述约束，可以让当前代码自然演进到蓝图中描述的形态，而不需要大规模重构。

## 6. 配置格式支持策略

为避免核心复杂度失控，同时兼顾未来演进空间，推荐采用“**JSON/INI 内核 + 多格式适配层**”的总体策略：

### 6.1 分级支持模型

1. **强支持（完整编辑 + 验证）**  
   仅限 INI + JSON：
   - 拥有完整的类型系统（`TConfigType` / `TEditorType` / 复杂属性枚举）；
   - 有专门的可视化编辑器 Frame 与验证规则；
   - 是代码与文档的“首等公民”。

2. **中等支持（导入 / 导出 + 转 JSON）**  
   适用于 YAML / TOML / 部分约定良好的 XML：
   - 在导入时解析为内部 JSON 结构（尽量保留注释/顺序等元信息）；
   - 经过编辑器在 JSON 模型上完成全部编辑与验证；
   - 保存时再转换回原始格式输出；
   - UI 始终围绕 JSON/INI，不单独维护“YAML 编辑器”等新 UI 栈。

3. **弱支持（只读预览 / 大文本字段）**  
   适用于架构不清晰或结构过于复杂的格式：
   - 可以作为“外部文件链接”或“大文本字段”挂在 JSON 中；
   - 编辑器只提供文件选择与只读预览，不尝试做结构化编辑和验证。

### 6.2 针对常见格式的具体建议

- **YAML / TOML**：  
  - 不在内核中直接支持其语义，而是实现 YAML/TOML ↔ JSON 的导入/导出适配；  
  - 保证所有类型/验证/编辑器仍仅面向 JSON/INI。

- **XML**：  
  - 默认不做强支持；如确有需要，应先约定一个较小的 XML 子集，再通过适配层转换为 JSON；  
  - 避免把命名空间、复杂 Schema/DTD 规则直接引入内核类型系统。

- **.env / properties / 纯 key=value 文本**：  
  - 视作“单节 INI”（例如自动挂到一个隐含的 `[default]` 下），复用现有 INI 管线；  
  - 不再额外设计一套新的格式模型。

### 6.3 与现有实现的关系

- JSON/INI 仍是唯一的内部事实来源；
- 新格式只通过“导入到 JSON/INI → 编辑 → 导出”的方式参与流程；
- UI、验证、复杂属性编辑器均不依赖具体物理格式，而是依赖统一的抽象模型。

这意味着即便以后引入 YAML/TOML/XML 支持，也不会破坏当前代码的核心结构，只是在边缘增加解析与序列化模块即可。
