# ConfigBuild 技术栈与开发约定

> 适用范围：本文件描述 **ConfigBuild（通用配置编辑器）** 的技术栈选择和开发约束，
> 面向参与本项目开发的程序员与 AI 辅助工具。

## 1. 概述

ConfigBuild 是一个基于 **Delphi + FMX** 的跨平台桌面应用，
用于可视化编辑“INI + JSON 双文件”配置，并通过 UniBase 提供统一的配置、日志和国际化能力。

本文件给出：

- 本项目必须使用的技术栈；
- ConfigBuild 内部的模块分层；
- 开发时必须遵守的关键约束与禁止事项。

## 2. 技术栈总览

### 2.1 核心语言与框架

- 语言：Delphi（Object Pascal）。
- UI 框架：**FireMonkey (FMX-only)**。
- 目标平台：优先支持 Windows，整体设计保持跨平台。

### 2.2 运行时与 UI 控件

- Delphi RTL：`System.*`、`System.IniFiles`、`System.JSON` 等。
- FMX 控件：
  - 布局与容器：`TLayout`, `TPanel`, `TSplitter`；
  - 数据展示：`TTreeView`, `TStringGrid`/`TGrid`；
  - 输入控件：`TEdit`, `TComboBox`, `TCheckBox`, `TTrackBar`, `TColorComboBox`；
  - 文本预览：**`TMemo`**（用于显示/编辑原始 INI / JSON 文本，以及简易帮助说明）。

### 2.3 配置与数据访问

- 被编辑目标配置（外部程序的配置）：
  - INI + JSON 双文件模型；
  - 通过 `INIConfig.pas` / `JSONConfig.pas`（或等价单元）统一封装；
  - 由 `ConfigManager` 负责加载、保存和一致性维护。

- 工具自身配置（ConfigBuild 的状态与偏好）：
  - 使用 UniBase 的 `config.db`（SQLite）+ `UniBase.Config` 存储；
  - 不在本项目内部再新建 INI / Registry / JSON 作为“内部配置源”。

## 3. 必须使用的 UniBase 模块

ConfigBuild 必须依赖并复用以下 UniBase 模块：

- `UniBase.Manager`
  - 初始化框架（`UniBase.Initialize` / `Finalize`）；
  - 解析 `root.txt`，定位 `{AppName}Config.db`；
  - 提供 `RootPath`、`ConfigDBPath` 等基础信息。

- `UniBase.Config`
  - 统一读写应用自身配置：
    - 最近打开工程列表；
    - UI 语言、主题、缩放比例；
    - 窗口位置/大小等。

- `UniBase.Logging`
  - 记录关键操作、错误、验证结果摘要等；
  - 所有异常处理路径必须至少写一条 Error / Fatal 日志。

- `UniBase.i18n`
  - 所有用户可见文本必须通过 i18n 资源管理，不允许硬编码中文/英文；
  - 使用 `T()` / `TFmt()` 等函数读取文本。

- `UniBase.FormState`（或等价 FMX 版本）
  - 自动保存/恢复主窗口及关键子窗口的状态。

- `UniBase.Security`
  - 如需存储敏感信息（例如外部工具的 API Key），必须通过 `TSecrets.SaveSecret/LoadSecret` 使用 DPAPI + `Secrets` 表加密存储。

### 3.1 UniBase FMX 支持确认

UniBase 已提供完整的 FMX 控件库（位于 `UniBase/FMX/` 目录），ConfigBuild 可直接复用：

- `UniBase.FMX.ConfigControls.pas` / `UniBase.FMX.ConfigEdit.pas`：配置绑定控件；
- `UniBase.FMX.I18nControls.pas`：国际化标签、按钮等；
- `UniBase.FMX.FormStateHelper.pas`：窗体状态自动保存/恢复；
- `UniBase.FMX.MRUControls.pas`：最近使用文件列表控件；
- `UniBase.FMX.Theme.pas`：主题切换支持；
- `UniBase.FMX.WaitForm.pas`：等待对话框；
- `UniBase.FMX.Dialogs.pas`：通用对话框封装；
- `UniBase.FMX.ListView.pas`：增强列表视图。

开发时优先使用 UniBase FMX 控件，避免重复封装。

## 4. ConfigBuild 内部模块分层

整体分层结构：

```text
FMX View (窗体/Frame)
        ↓
Controller（流程控制/用户动作编排）
        ↓
Config 核心（ConfigManager / ConfigValidator / ControllerConfigs 等）
        ↓
UniBase 基础设施（Config / Logging / i18n / Security）
        ↓
文件系统（INI + JSON）、SQLite config.db
```

### 4.1 View 层（FMX 界面）

- 单元命名：`View*.pas` 或语义清晰的 FMX Form/Frame 单元。
- 只负责：
  - 布局和显示（树、网格、文本预览、按钮等）；
  - 收集用户输入；
  - 调用 Controller / 核心模块公开的接口；
  - 根据返回结果更新 UI（文本、颜色、高亮）。
- 禁止：
  - 直接读写 INI / JSON 文件；
  - 直接操作 UniBase.Config / Logging（除非常简单的日志）以外的底层细节；
  - 在事件处理函数中堆叠复杂业务逻辑和判断。

### 4.2 Controller 层

- 单元命名示例：`CtrlConfigEditor.pas`、`CtrlValidatorRunner.pas` 等。
- 职责：
  - 把用户动作编排成高层操作流程（打开、验证、保存等）；
  - 调用 `ConfigManager`、`ConfigValidator` 等核心模块；
  - 处理错误并将结果反馈给 View（以简单的状态对象/错误列表形式）。

### 4.3 核心模块层

- `ConfigManager`：
  - 读写 INI + JSON；
  - 统一入口：打开工程、保存工程、备份策略。

- `ConfigValidator`：
  - 配置验证规则和执行；
  - 输出验证结果列表（错误/警告/自动修复建议等）。

- `ControllerConfigs`：
  - 根据配置类型选择并创建 `TBaseConfigFrame` 子类；
  - 管理复杂属性编辑器的加载/保存流程。

- `UtilsTypes`：
  - 定义配置类型系统（`TConfigType` / `TEditorType` 等）；
  - 为“单一事实来源”。

## 5. 开发约定与禁止事项

### 5.1 必须遵守

- **FMX-only**：
  - 不在本项目中使用 VCL 控件和 VCL-only 组件；
  - 跨平台相关代码不得使用 Win32 特有 API，除非封装在可替换的适配层中。

- **配置单一来源**：
  - 外部业务配置：INI + JSON 文件，对应 MVP 文档；
  - 工具自身配置：UniBase.Config；
  - 禁止再新增独立 INI / Registry / JSON 作为新配置源。

- **文案与国际化**：
  - 所有用户可见文本通过 UniBase.i18n 管理；
  - 禁止硬编码中文或英文提示；
  - 带参数文本使用 `TFmt()`，避免字符串拼接。

- **日志与错误处理**：
  - 关键操作必须记录 Info / Warn / Error 日志；
  - 捕获异常时，不吞掉详细信息，至少写入一条包含异常消息的日志。

### 5.2 严格禁止

- 直接使用 `TIniFile`/`Registry` 保存工具自身配置；
- 在 View 层直接操作文件系统或 JSON 结构；
- 在任何地方硬编码敏感信息（密码、Token 等）；
- 在事件处理中编写长逻辑而不抽取到可测试的核心模块。

## 6. 新功能开发 Checklist

在 ConfigBuild 中新增或修改功能时，应至少自查：

1. 是否复用了现有模块（ConfigManager / ConfigValidator / UniBase.*），而不是重复造轮子？
2. 新增的配置字段是否已经：
   - 在 `UtilsTypes` 中定义类型；
   - 在相应模型文档中补充说明；
   - 在验证逻辑中考虑？
3. UI 是否仅做展示与输入，没有直接访问底层文件 / 数据库？
4. 所有新文本是否接入 i18n？
5. 是否在关键路径增加了日志？
6. 是否在 `docs/` 中补充或更新了对应文档（产品 / 模型 / 架构 / 使用 / 测试）？

## 7. UI 增强计划（待实现）

以下 UI 增强功能已规划，开发时应逐步纳入：

### 7.1 状态栏

- 位于主界面底部，提供轻量级反馈：
  - 当前文件路径和修改状态（已修改 / 已保存）；
  - 验证结果摘要（如"3 个错误 / 2 个警告"）；
  - 最近操作提示（如"配置已保存"）。

### 7.2 复杂属性区域空状态

- 当未选中复杂属性时，右侧 Frame 区域显示占位提示；
- 可考虑支持折叠，节省屏幕空间。

### 7.3 JSON 数组拖拽排序

- 对于 JSON 数组节点（如多个 AI Provider、多个数据库连接），支持拖拽调整顺序；
- 在树视图或专用列表中实现。

### 7.4 上下文感知编辑

- Frame 标题或工具栏显示当前正在编辑的属性路径（如 `AI.DefaultProvider`）；
- 避免用户在多个复杂属性间切换时迷失上下文。

## 8. 示例配置文件

项目 `configs/` 目录包含示例配置文件，用于开发和测试：

- `sample.ini`：演示 INI 文件中的简单属性（字符串、整数、布尔值）；
- `sample.json`：演示 JSON 文件中的复杂属性类型（Font、Database、AIAPI、BgDraw、VideoClip、Video）。

开发新功能时，应同步更新示例文件以覆盖新增的属性类型。
