# ConfigBuild 整体架构与模块划分

> 本文描述 ConfigBuild（通用配置编辑器）的整体架构、主要模块与依赖关系，
> 作为理解代码结构和扩展点的入口文档。

## 1. 架构概览

ConfigBuild 是一个基于 **Delphi + FMX** 的跨平台桌面应用，用于可视化编辑 INI + JSON 双文件配置。
系统整体分为四层：

```text
FMX View（窗体 / Frame）
        ↓
Controller 层（用户操作编排）
        ↓
配置核心层（ConfigManager / ConfigValidator / ControllerConfigs / UtilsTypes）
        ↓
基础设施层（UniBase.* + 文件系统 + SQLite）
```

## 2. 分层说明

### 2.1 View 层（FMX 界面）

- 典型单元：`ViewMainConfigEditor.pas`（示例命名）、各类 `TFrame`。
- 职责：
  - 展示配置树、属性网格、原始文本和验证结果；
  - 响应用户操作（按钮点击、树节点选择等）；
  - 调用 Controller 层提供的接口。
- 不负责：
  - 文件读写、JSON 解析、验证规则等核心逻辑。

### 2.2 Controller 层

- 典型单元：`CtrlConfigEditor.pas`、`CtrlValidateConfig.pas`（示例）。
- 职责：
  - 将用户的高层操作（打开、验证、保存）映射到配置核心层；
  - 把核心层返回的结果（错误列表、修改后的配置）整理为 View 可消费的形式；
  - 作为 View 与核心层之间的防护墙。

### 2.3 配置核心层

- 核心单元及职责：
  - `ConfigManager.pas`：
    - 统一处理 INI + JSON 文件加载与保存；
    - 管理备份策略（例如 `.bak + 时间戳`）；
    - 提供“打开工程 / 保存工程”等高层接口。
  - `ConfigValidator.pas`：
    - 定义和执行配置验证规则；
    - 输出错误 / 警告 / 自动修复建议等结构化结果。
  - `ControllerConfigs.pas`：
    - 根据配置类型和上下文选择合适的复杂属性编辑器 Frame；
    - 管理 `TBaseConfigFrame` 子类的创建与销毁。
  - `UtilsTypes.pas`：
    - 定义 `TConfigType`、`TEditorType` 及复杂属性相关枚举；
    - 作为整个项目的“类型系统单一事实来源”。

### 2.4 基础设施层

- UniBase：
  - `UniBase.Manager`：初始化、RootPath / ConfigDBPath 解析；
  - `UniBase.Config`：应用自身配置；
  - `UniBase.Logging`：日志系统；
  - `UniBase.i18n`：国际化；
  - `UniBase.Security`：敏感信息存储。
- 文件系统：
  - 目标 INI + JSON 配置文件存放位置；
  - 备份文件目录（如 `backup/`）。
- SQLite `config.db`：
  - 存储 UniBase 自身表和 ConfigBuild 的内部配置。

## 3. 典型调用流程

### 3.1 打开配置工程

1. 用户在 View 中选择某个 INI 文件。
2. Controller 调用 `ConfigManager.OpenProject(IniPath)`。
3. `ConfigManager`：
   - 解析 INI，找到关联 JSON 路径；
   - 加载 JSON 文本并解析为内部对象；
   - 构建用于 UI 展示的中间模型（树节点、属性列表等）。
4. Controller 将模型传递给 View，View 刷新树和网格。

### 3.2 编辑与验证

1. 用户在网格 / Frame 中修改配置值。
2. View 将变更通知 Controller，Controller 更新内部模型或直接调用核心层接口。
3. 用户点击“验证”按钮：
   - Controller 调用 `ConfigValidator.Validate(AllConfig)`；
   - `ConfigValidator` 返回验证结果列表；
   - Controller 将结果转为 UI 所需格式（如每条错误对应的路径和消息）。
4. View 根据验证结果高亮相关节点或行，并在底部日志区显示摘要。

### 3.3 保存与备份

1. 用户点击“保存”：
   - Controller 调用 `ConfigManager.SaveProject`；
2. `ConfigManager`：
   - 根据策略生成备份文件（如 `.bak`）；
   - 先写 JSON，再写 INI，或按约定顺序统一写入；
   - 出错时抛出异常或返回错误码，由 Controller 处理并提示用户。
3. 成功保存后，通过 `UniBase.Logging` 记录 Info 级日志。

## 4. 关键单元一览（示例）

> 具体单元名以实际代码为准，以下为推荐/典型划分。

- View 层：
  - `ViewMainConfigEditor.pas`：主界面窗口；
  - `FrameSimplePropertyEditor.pas`：简单属性编辑 Frame；
  - 多个复杂属性编辑 Frame：`FrameFontEditor.pas`、`FrameDBEditor.pas` 等。

- Controller 层：
  - `CtrlConfigEditor.pas`：整体配置编辑控制器；
  - `CtrlValidateConfig.pas`：验证流程控制器。

- 配置核心层：
  - `ConfigManager.pas`、`ConfigValidator.pas`、`ControllerConfigs.pas`、`UtilsTypes.pas` 等。

- 基础设施：
  - UniBase 相关单元（位于独立 UniBase 仓库中，此处仅引用）。

## 5. 扩展点

### 5.1 新增复杂属性类型

1. 在 `UtilsTypes.pas` 中新增对应的配置类型枚举值；
2. 新建一个继承自 `TBaseConfigFrame` 的 FMX Frame 单元，实现：
   - `LoadFromJSON` / `SaveToJSON`；
   - 必要的输入校验和 UI 交互；
3. 在 `ControllerConfigs.pas` 中注册类型与 Frame 之间的映射；
4. 在文档 `02.02.模型-复杂属性类型与编辑器映射-v1.0.md` 中补充说明；
5. 增加至少一个示例配置和测试用例。

### 5.2 新增验证规则

1. 在 `ConfigValidator.pas` 中新增规则实现（或配置驱动规则）；
2. 为该规则设计最小可重现的配置样例；
3. 更新测试文档和测试用例，确保回归场景覆盖；
4. 在 UI 文案中增加清晰的错误提示（通过 i18n 管理）。
