# ConfigBuild 测试策略与回归清单

> 本文描述 ConfigBuild（通用配置编辑器）的测试目标、测试方式和基本回归场景，
> 用于指导人工测试和后续自动化测试的设计。

## 1. 测试目标

- 确保在常见场景下，ConfigBuild 能稳定完成：
  - 打开 INI+JSON 配置对；
  - 浏览和编辑简单/复杂属性；
  - 执行验证并给出正确提示；
  - 安全地保存和备份配置文件。
- 避免常见破坏性错误：数据丢失、保存到错误位置、产生无法解析的 JSON 等。

## 2. 测试类型

### 2.1 手工测试（主方式）

- 通过主程序和示例配置进行人工操作测试；
- 重点覆盖典型用户路径和复杂属性编辑流程。

### 2.2 示例工程 / 测试项目

- `tests/` 目录下包含若干 `.dpr` 和 `.pas` 文件，用于：
  - 验证特定配置结构（如 `MediaConfig.pas`, `SystemConfig.pas`, `UIConfig.pas`）；
  - 演示或验证特定编辑器行为（如 `SimpleConfigEditor.dpr`, `FinalConfigEditor.dpr`）。
- 这些工程可以作为“测试沙箱”，在其中加载不同的配置样例进行测试。

### 2.3 单元测试（后续扩展）

- 推荐为关键核心模块（`ConfigManager`, `ConfigValidator`, `UtilsTypes` 等）逐步增加 DUnitX 单元测试；
- 测试工程可以单独放在 `Tests/` 目录下（与示例工程区分）。

## 3. 基本回归场景

### 3.1 打开与关闭

1. 启动 ConfigBuild；
2. 打开一个合法的 INI（带合法 JSON 引用）；
3. 确认：
   - 左侧树正确展示 INI 节和 JSON 对象；
   - 中间视图中能看到网格和文本预览；
4. 关闭工程或退出程序，无异常弹窗。

### 3.2 简单属性编辑

1. 打开示例配置；
2. 修改几个简单属性：字符串、整数、布尔、颜色等；
3. 触发验证，确认无严重错误；
4. 保存并关闭程序；
5. 使用文本编辑器查看 INI/JSON，确认修改被正确写入且格式未损坏。

### 3.3 复杂属性编辑

1. 打开包含字体、数据库、背景/视频等复杂属性的示例配置；
2. 分别打开各类复杂属性编辑器 Frame：
   - 修改一些字段并保存；
   - 再次打开确认修改被正确加载；
3. 触发验证，确认：
   - 缺失必填字段时会给出错误；
   - 范围错误（如时间超出视频长度）会给出警告或错误。

### 3.4 验证与错误提示

1. 打开一个故意带有错误的配置（缺字段、错误路径、无效引用等）；
2. 触发验证，检查：
   - 错误数量和类型是否符合预期；
   - 日志区信息是否清晰可读；
   - 是否能通过点击或双击快速定位到出错项（如实现）。

### 3.5 保存与备份

1. 在编辑配置后点击“保存”；
2. 确认：
   - `backup/` 目录（或指定位置）是否生成备份文件；
   - 主 INI 和 JSON 文件是否成功写入；
3. 模拟保存过程中出现异常（如路径不可写），检查：
   - 是否给出明确错误提示；
   - 是否避免产生半写入/损坏文件。

## 4. 测试数据与样例

- 建议在 `configs/` 或 `tests/` 下维护一组标准测试配置：
  - 正常配置样例（无错误）；
  - 带轻微警告的样例（如推荐值范围外但可自动修复）；
  - 带严重错误的样例（缺失必填字段、JSON 结构错误等）。
- 每个样例应附简短说明（可在 `99.01.归档-历史文档与命名映射-v1.0.md` 或专门说明文件中记录）。

## 5. 新功能的测试要求

每当在 ConfigBuild 中新增或修改功能时，应至少：

1. 明确该功能影响的配置区域（INI / JSON 哪一部分）；
2. 增加或更新对应的测试样例；
3. 按第 3 节中的相关回归场景执行一次完整测试；
4. 如修改涉及核心模块（加载/保存/验证），优先考虑增加单元测试；
5. 在提交前检查日志和异常输出，确保无明显未处理错误。
