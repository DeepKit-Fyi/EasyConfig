# ConfigBuild 开发文档命名与结构指南

> 适用范围：本指南只针对 **ConfigBuild 仓库** 的 `docs/` 目录，不影响其它项目。
> 编写任何新文档前，应先阅读本文件和《Delphi 程序开发约定指南》的“项目文档结构”章节。

## 1. 目标与原则

- 为 ConfigBuild 建立统一的文档命名和结构规范。
- 控制文档数量：**少而精**，优先补充现有文档，而不是新建相似文档。
- 所有文档必须：
  - 聚焦“产品内核”和“可执行步骤”；
  - 避免时间线描述（不写具体排期和里程碑日期）。

## 2. 文件命名规则

统一命名格式：

`xx.yy.{分类名}-{文件名}-{版本号}.md`

### 2.1 字段说明

- `xx`：分类编号（01–99，两位数字）。
- `yy`：同一分类下的小序号，从 01 开始递增。
- `{分类名}`：中文分类名称，例如：`产品`、`模型`、`架构`、`使用`、`规范`、`测试`、`归档`。
- `{文件名}`：中文主题名称，简短清晰，如：`通用配置编辑器MVP产品构成`。
- `{版本号}`：`v主版本.次版本`，例如：`v1.0`、`v1.1`。

### 2.2 示例

- `01.01.产品-通用配置编辑器MVP产品构成-v1.0.md`
- `01.02.产品-通用配置编辑器未来产品蓝图-v1.0.md`
- `05.01.规范-ConfigBuild开发文档命名与结构指南-v1.0.md`

> 说明：旧文档暂时可以保留旧命名；新文档必须使用本规范，旧文档在后续整理时重命名或合并。

## 3. 分类编号表

ConfigBuild 仓库推荐使用以下分类编号：

- **01 产品**：产品定位、MVP 范围、未来蓝图。
- **02 模型**：INI+JSON 配置模型、类型系统、复杂属性结构。
- **03 架构**：整体架构、模块划分、关键单元说明。
- **04 使用**：面向用户的操作手册和 FAQ。
- **05 规范**：开发规范、文档规范、本仓库特有约束。
- **06 测试**：测试策略、回归清单、测试数据说明。
- **99 归档**：历史文档、一次性记录、命名映射等。

不建议新增新的 `xx`，除非现有分类确实无法覆盖。

## 4. docs 目录结构

- 所有文档统一放在 `docs/` 目录下。
- 不再按子目录划分逻辑分类，分类完全依赖文件名前缀 `xx.yy.{分类名}-`。
- UI 草图、界面设计等放在 `docs/ui/` 子目录中（例如：`docs/ui/通用配置编辑器布局.svg`）。

示例：

- `docs/01.01.产品-通用配置编辑器MVP产品构成-v1.0.md`
- `docs/01.02.产品-通用配置编辑器未来产品蓝图-v1.0.md`
- `docs/02.01.模型-INI与JSON配置结构与约束-v1.0.md`
- `docs/03.01.架构-ConfigBuild整体架构与模块划分-v1.0.md`
- `docs/04.01.使用-通用配置编辑器操作手册-v1.0.md`
- `docs/05.01.规范-ConfigBuild开发文档命名与结构指南-v1.0.md`
- `docs/05.02.规范-ConfigBuild技术栈与开发约定-v1.0.md`
- `docs/06.01.测试-ConfigBuild测试策略与回归清单-v1.0.md`
- `docs/99.01.归档-历史文档与命名映射-v1.0.md`

## 5. 文档内容结构模板

所有新文档推荐遵循如下结构（可按需要精简）：

```markdown
# 文档标题

> 文档版本：vX.Y  
> 更新日期：YYYY-MM-DD  
> 适用范围：说明本文件作用

## 1. 概述
- 说明文档类型（产品 / 模型 / 架构 / 使用 / 规范 / 测试）。
- 说明目标读者和要解决的问题。

## 2. 产品内核 / 核心概念
- 列出与主题相关的核心概念、约束和价值。

## 3. 实现 / 使用步骤
1. 步骤一：...
2. 步骤二：...
3. 步骤三：...

## 4. 关联代码与文档
- 相关单元：...
- 相关文档：...

## 5. 注意事项 / 常见错误
- 列出必须遵守的约束与典型坑点。
```

## 6. “不写时间线”的要求

- 文档中不写项目排期和具体日期的计划安排。
- 允许在开头写“文档版本 / 更新日期”，它只是元信息，不是计划时间线。
- **错误示例（禁止）：**

```markdown
## 开发计划
- 2025-12-01: 完成配置加载模块
- 2026-01-01: 上线
```

- **正确示例（推荐）：**

```markdown
## 实现步骤
1. 完成配置加载与保存模块。
2. 完成验证模块并与 UI 集成。
3. 增加测试用例和回归测试入口。
```

## 7. 与《Delphi 程序开发约定指南》的关系

- 通用的命名规范、架构分层、国际化、性能优化等内容，以
  《Delphi 程序开发约定指南》为唯一事实来源，不在 ConfigBuild 文档中重复大篇幅说明。
- ConfigBuild 文档只描述：
  - 本仓库特有的约束；
  - 与 UniBase 集成方式；
  - 本项目的配置模型和 UI 设计。

## 8. 新增 / 修改文档流程

1. 判断新内容属于哪一类（产品 / 模型 / 架构 / 使用 / 规范 / 测试 / 归档）。
2. 在对应分类下选择下一个可用的小序号 `yy`，构造文件名：
   `xx.yy.{分类名}-{文件名}-{版本号}.md`。
3. 使用第 5 节的模板编写内容，避免时间线，突出产品内核和可执行步骤。
4. 如要修改现有文档：
   - 小改动：可以保持文件名不变，仅在文档头部更新版本号和更新日期；
   - 大改动：建议同时提升 `{版本号}`（如从 `v1.0` 升级为 `v1.1`）。
