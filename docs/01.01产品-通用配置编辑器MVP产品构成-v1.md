# 01.01 通用配置编辑器 MVP 产品构成

## 1. 产品定位

通用配置编辑器的目标，是为**桌面应用和服务端程序**提供一套统一的「配置文件可视化编辑工具」，解决如下痛点：

- 配置分散在 INI / JSON 等多种格式中，结构复杂、不易理解；
- 业务开发人员/运维需要频繁修改配置，但不熟悉具体格式细节，容易出错；
- 需要一定程度的「结构化约束」与「验证规则」，而不是纯文本编辑。

本工具围绕“**INI + JSON 双文件配置模型**”展开：

- 简单的键值配置存储在 INI 中；
- 复杂对象、数组、背景/视频/AI 模型等结构化配置存储在 JSON 中；
- 由编辑器统一展示、编辑，并保证两者之间的一致性；
- 验证逻辑可抽象为规则（`ConfigValidator`），与 UI 解耦。

> 换句话说：这是一个面向开发者/高级用户的 **配置建模与编辑器**，而不是终端用户向导。

---

## 2. 典型用户与使用场景

### 2.1 典型用户

- **应用开发者**：
  - 为自己的程序设计配置结构；
  - 通过本工具维护默认配置和测试配置；
  - 给同事/客户一个安全的编辑入口。
- **实施/运维工程师**：
  - 按项目实际部署环境（测试 / 预发 / 生产）调整配置；
  - 在出现问题时排查配置是否有误、是否缺字段。

### 2.2 典型使用场景

- 打开某个工程的 INI + JSON 配置对（如：`app_config.ini` + `app_config.json`）。
- 在左侧树中浏览配置结构（逻辑分组、复杂对象、数组条目等）。
- 在中间区域编辑 INI/JSON 对应的视图（网格 + 树 + 文本预览）。
- 通过专用编辑器 Frame 修改复杂属性（字体、数据库、AI 模型、背景布局、视频片段等）。
- 点击“验证”按钮，查看规则校验结果并根据提示修复错误。
- 保存后，将修改写回对应的 INI 与 JSON 文件，为业务程序下一次启动/热加载使用。

## 3. MVP 的核心能力

MVP 版本只追求“**可以安全地编辑一套 INI+JSON 配置，并完成基本验证**”，不追求所有高级功能一次到位。

### 3.1 配置模型与类型系统

- 支持 **INI + JSON 一对一** 的配置单元：
  - INI 文件中通过固定节（如 `[json_file]` / `file_path`）指向 JSON 文件；
  - JSON 文件中通过 `_type`、`_id` 等字段标记对象类型与标识。
- 使用统一的 **配置类型枚举**（`TConfigType`）和 **编辑器类型枚举**（`TEditorType`）：
  - 简单类型：文本、整数、浮点、布尔、颜色、路径、列表等；
  - 复杂类型：字体、数据库连接、AI API、背景/图像/文本叠加、视频片段、工作流等；
  - 通过 `UtilsTypes.pas` 提供统一的类型转换与推断逻辑。

### 3.2 浏览与打开配置

- 通过文件对话框选择 INI 文件，自动解析其关联 JSON 文件路径：
  - 若 JSON 不存在，可按约定创建空 JSON；
  - 支持相对路径与绝对路径。
- 在 UI 中展示：
  - **INI 视图**：使用 `TStringGrid` 以“属性名 / 属性类型 / 属性值”形式展示；
  - **JSON 视图**：使用 `TTreeView` 展示对象树结构，并在节点上挂载元信息。

### 3.3 简单类型的可视化编辑

MVP 需要完整支持以下简单配置类型：

- 文本、整数、浮点、布尔；
- 路径/文件名（相对/绝对路径转换由工具负责）；
- 颜色（通过颜色对话框选择）；
- 枚举/列表（在 UI 中提供下拉框或列表选择）。

具体交互形式：

- 通过添加按纽（“添加文本属性”、“添加数值属性”等）在网格中新建行；
- 双击单元格或通过右键菜单编辑已有属性；
- 支持修改键名、键值，并自动同步展示到 INI 文本区域。

### 3.4 复杂类型的可视化编辑

MVP 中只要求支持**一部分优先级高的复杂类型**，并通过专用 Frame 编辑：

- 字体（Font）
- 数据库连接（Database）
- AI 大模型 API（AIAPI）
- 背景 + 文本 + 图像（BgDraw / TextOnBg / ImageOnBg）
- 视频片段 / 视频组合（VideoClip / Video）

核心机制：

- 每个复杂属性对应一个 `TBaseConfigFrame` 子类（如 `FrameFontEditor`、`FrameDBEditor`）；
- 由 `ControllerConfigs` 根据按钮 Tag / `TComplexPropertyType` 决定创建哪个 Frame；
- Frame 读写的对象为某个 JSON 子对象（`JSONObject`），并实现 `LoadFromJSON` / `SaveToJSON`；
- 编辑完成后，将 JSON 结果返回给主界面，并写回 JSON 文件。

### 3.5 验证与错误提示

- 使用 `ConfigValidator` 提供可组合的验证规则：
  - 必填、数值范围、字符串长度、正则、Email、URL、IP、日期时间等；
  - 支持部分字段的 AutoFix（如去除非法字符、剪裁范围、补全 URL 协议等）。
- 在 `ViewBuildConfig` 中提供：
  - “验证”按钮触发全局验证；
  - 验证结果通过对话框/底部日志区展示，并（计划）高亮对应单元格/树节点。

### 3.6 保存与备份

- 同时保存 INI 和 JSON：
  - 先写 JSON，再写 INI，或通过 `ConfigManager` 统一调度；
  - 必要时提供简单备份策略（例如 `.bak + 时间戳`）。
- MVP 阶段可以只支持手动保存（点击按钮），自动保存作为后续版本考虑。

## 4. MVP 的产品边界

为了控制复杂度、确保短期内可交付，MVP 明确**不做**以下事情：

### 4.1 不执行业务逻辑 / 不渲染实际效果

- 不真正连接数据库、执行 SQL；
- 不真正调用 AI API、发送 HTTP 请求；
- 不负责渲染视频、音频或实时画面预览；
- 不负责渲染完整 UI 主题，仅编辑其配置参数。

> 工具的职责是**生成 / 修改配置文件**，业务含义和运行时行为由外部程序负责。

### 4.2 不做运行时托管与热更新

- 不做「监控目标程序并自动 reload 配置」这类运行时控制；
- 不管理进程生命周期，不是进程管理器或服务管理器；
- 不提供远程配置下发、集中配置中心等分布式能力（仅操作本地文件）。

### 4.3 不做用户/权限与协作

- 不实现用户登录、权限隔离、操作审计；
- 不处理多人并发编辑、合并冲突；
- 不内置工作流（审核、发布）能力。

这些能力在未来蓝图中可以作为“团队版/企业版”的方向，但不属于 MVP 范围。

### 4.4 不作为通用 IDE 或代码编辑器

- 不编译、不运行业务程序；
- 不提供代码补全、调试器、版本控制 UI 等开发环境功能；
- 只是一个偏「领域化」的配置设计与编辑工具，可以和 IDE 协同使用。

### 4.5 文件与格式边界

- MVP 只面向：
  - Windows 桌面环境；
  - INI + JSON 这一对格式；
- 不支持 YAML、XML、TOML 等其他配置格式（可以在 JSON 中嵌入部分这类内容，但编辑器不理解其结构）。
- 如确有需要，可以在外部工具中将 YAML / TOML / XML 等转换成 JSON 后再导入编辑器，或作为纯文本字段嵌入 JSON；
- 其它格式的正式支持策略见《01.02.通用配置编辑器未来产品蓝图》中的“配置格式支持策略”一节。

## 5. 与其他工具的关系

- 与 **纯文本编辑器**（如 Notepad++）：
  - 本工具提供类型感知、结构化 UI、验证规则；
  - 文本编辑器则适合快速查看/批量替换，两者互补。

- 与 **业务程序**：
  - 配置编辑器负责生成/维护配置；
  - 业务程序在运行时加载这些配置并决定具体行为；
  - 当业务程序升级需要新字段时，应先更新类型定义与模板，再在配置编辑器中维护。

- 与 **未来的配置中心/云平台**：
  - MVP 仅处理本地文件，但设计上尽量使 JSON/INI 模型、验证逻辑、复杂属性结构可移植到未来的集中式配置平台中。
